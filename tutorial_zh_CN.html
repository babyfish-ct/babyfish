Java部分

Java部分用于支持智能数据结构，同时它也是JPA部分的基础

核心功能： ObjectModel4Java

ObjectModel4Java是BabyFish框架的纯Java部分的最上层的功能，协调Java数据对象之间的所有双向关系，包括一对一、一对多和多对多关系。此处仅以List集合和IndexedReference引用形成的一对多双向关系为例，展现ObjectModel4Java的魅力。

以经典的部门和员工之间的一对多关系为例。一个部门可以拥有多个员工，在本例中我们要求部门使用有索引顺序的集合（即java.util.List）来维护多个员工对象；反过来对于每一个员工对象而言，本例不但要求它能反向保持所属部门的父对象引用，而且还要求它能保持它在其所属部门中的索引位置。

  • 父对象Department的类定义 
    @Model // 注解@org.babyfish.model.Model表示这不是普通的数据类，要求BabyFish的Maven插件在编译时对其进行字节码增强
public class Department {

    @Association(opposite = "department")// 注解@Association声明关联属性"employees"，和Employee.department互为镜像
    private List<Employee;> employees;

    //获取Employee子对象的关联集合，因为集合本身可变，故无需setter（当然，提供setter也没错）
    public List<Employee> getEmployees() {
        return this.employees; 
    }
}
  • 子对象Employee的类定义 
    @Model // 注解@Model表示这不是普通的数据类，要求BabyFish的Maven插件在编译时对其进行字节码增强
public class Employee {
    
    @Association(opposite = "employees")// 注解@Association声明关联属性"department"，和Department.employees互为镜像
    private Department department;

    @IndexOf("department")// 注解@IndexOf表示该索引是当前对象在department字段所指父对象的employees集合中的位置(无父对象时为-1)
    private int index;

    public int getIndex() {
        return this.index; 
    }

    public void setIndex(int index) {
        this.index = index; 
    }

    public Department getDepartment() {
        return this.department;
    }

    public void setDepartment(Department department) {
        this.department = department;
    }
}

上述代码中，Department.employees和Employee.department互为镜像，这是表示Department和Employee之间的关联是双向的。BabyFish保证这对双向关系的一致性，用户既可以使用Department端修改关联也可以使用Employee端修改关联，任何一端的修改都会触发事件通知另外一端使其自动做出相应的同步修改。

不稳定集合元素
为什么我们需要不稳定元素

几乎在所有语言的集合框架中都存在着Set和Map的数据数据结构（当然，某些语言称Map为Dictionary，如C#, Objective-C），它们是基于哈希结构或红黑树结构的，二者好处非常明显，能提供非常高的查询性能和不错的修改性能，但也有一个致命缺点，如果某个数据对象已经被添加进了哈希结构或红黑树结构，那么这个数据对象就不能被修改了，否则哈希结构或红黑树结构就会失效。Java语言也不例外，在Java Collection Framework中，Set和Map要求元素必须是稳定的。

这是因数据结构算法导致的限制，但在实际业务的数据建模过程中常常难以保证，带来麻烦。但是ObjectModel4Java可以轻松解决此问题，以基于ObjectModel4Java实现公司和投资者之间的双向多对多关系为例。

  • 公司类声明如下 
    @Model // 注解@Model表示这不是普通的数据类，要求BabyFish的Maven插件在编译时对其进行字节码增强
public class Company {

    @Scalar // @注解org.babyfish.model.Scalar表示name是一个普通标量属性，非关联属性
    private String name;
        
    /*
     * 一个公司可以由多个投资者合力投资。
     *
     * @Association注解：
     * (1) 本属性和Investor.companies属性互为镜像
     * (2) 本属性的集合接口为类型为普通Set，故此集合的行为类似于HashSet
     *

     * (1) 计算每个Investor对象的hashCode
     * (2) 判断两个Investor对象是否相等

    @Association(opposite = "companies")
    @ComparatorRule(properties = @ComparatorProperty(name = "name"))
    private Set<Investor> investors;

    public String getName() {
        return this.name; 
    }

    public void setName(String name) {
        this.name = name; 
    }

    //因为集合本身可变，故无需setter（当然，提供setter也没错）

上述数据结描述了公司和投资者之间的多对多关系。但有两个问题

  1. Company类的investors属性行为类似于java.util.HashSet，使用Investor类的name属性来计算hashCode以及判断两个Investor对象是否equals。但是，Investor类支持setName方法，其name属性是可以改变的，所以其hashCode是不稳定的，这显然不满足通常意义上所讲的哈希结构需要稳定元素的前提。
  2. Investor类的companies属性行为类似于java.util.TreeMap，使用Company类的name属性来比较两个Company对象的大小关系。但是，Company类支持setName方法，其name属性是可以改变的，所以不同的Company对象之间的大小关系是不稳定的，这显然不满足通常意义上所讲的红黑树结构需要稳定元素的前提。

为了避免这个头疼的问题，BabyFish基于Java Collection Framework的接口扩展出了X Collection Framework并给出了一整套全新的集合实现，让Set和Map突破了元素必须稳定这个难以保证的限制条件，即使某个数据对象已经被添加进了哈希结构或红黑树结构，该数据对象仍然可以被修改，因为引用该数据对象的所有哈希结构和红黑树结构均能随其修改而自动调整，这个神奇的功能就是不稳定集合元素。不稳定集合元素是X Collection Framework最重要的功能。因为不稳定集合元素功能的存在，上述的Company类和Investor类之间的双向多对多关系其实是可以正常工作的。

不稳定元素的幕后细节

由于ObjectModel4Java靠编译时植入额外的字节码的手段隐藏了所有的幕后动作，所以这一节暂时忽略ObjectModel4Java话题，借助更底层的X Collection Framework来论述此功能，把ObjectModel4Java在这方面所隐藏掉的一切呈现出来。

在讨论不稳定集合元素之前，需要先讲述org.babyfish.collection.EqualityComparator<T>接口。我们知道，Java集合TreeSet或TreeMap既可以通过元素类自身实现java.lang.Comparable<T>接口来指定元素比较规则，也可以直接用一个实现了java.util.Comparator<T>接口的第三方类来指定元素比较规则；而HashSet或HashMap却只能让元素类自身覆盖hashCode和equals方法来指定哈希码计算规则和相等性判断规则，无法使用第三方规则；为此，BabyFish框架借鉴了.NET的System.Collections.Generic.IEqualityComparer<T>接口，引入了org.babyfish.collection.EqualityComparator<T>接口解决此问题，此接口非常简单，查看源代码即可，这里不需论述。

  1. 为了让红黑树结构支持不稳定元素，BabyFish扩展了java.util.Comparator<T>接口 
    package org.babyfish.collection;

import java.util.Comparator<T>;

public interface FrozenComparator<T> extends Comparator<T> {

    // 当对象obj被添加到某个红黑树集合时，自动调用此方法冻结此对象
    void freeze(T obj, FrozenContext<T> ctx);

    // 当对象obj被某个红黑树集合移除时，自动调用此方法解冻此对象
    void unfreeze(T obj, FrozenContext<T> ctx);
}
  2. 同理，为了让哈希结构支持不稳定元素，BabyFish扩展了org.babyfish.collection.EqualityComparator<T>接口

implements FrozenComparator<FullName> {
    @Override

        Integer preCmp = org.babyfish.lang.Nulls.preCompare(o1, o2);
        if (preCmp != null) {
            return preCmp;
        }
        int cmp = org.babyfish.lang.Nulls.compare(o1.getFirstName(), o2.getFirstName());

元素排挤效应

不稳定元素还会导致一个有趣的问题现象，我们都知道，set（map）是不允许元素（键）重复的。如果用户企图修改某个元素（键）并使其值同另外一个元素（键）的值冲突，BabyFish集合框架将会怎么做？为了保证元素（键）的唯一性，两个冲突的元素（键）中必须有一个被排挤出当前集合，这就叫元素排挤行为，此行为有两种。

  1. 默认模式，保留当前用户修改的元素，自动删除另外一个冲突元素。在上层的ObjectModel中只能使用该模式。
  2. 第二种模式，保留另外一个冲突元素，自动删除当前用户修改的元素。这种模式无法在ObjectModel中使用，只能在底层的集合框架中使用，本文也深入不讨论之。

接着上一节的例子继续操作

// 修改fn2的值使其与fn3的值冲突，
//在默认的元素排挤模式下，fn3将会受到排挤，被集合自动删除。
fn2.setFirstName("gavin");
fn2.setLastName("king");

集合视图链之事件冒泡

上一章提到，为了支持以不稳定元素为主的一些功能，BabyFish框架从Java Collection Framework扩展出了X Collection Framework。为了进一步支持ObjectModel，BabyFish从XCollection Framework扩展出了MA Collection Framework。MA即ModificationAware，修改可感知的意思，表示集合框架支持修改行为的事件通知，和数据库的行级触发器非常类似。

MA Collection Framework能像数据库触发器一样触发前置事件和后置事件，这个很好理解，其功能点很多，没要全部在本文中讨论。本文仅讨论这部分最重要的功能：视图链事件冒泡。

除了语言本身使用擦除式泛型导致的硬伤以外，Java集合框架远远比其它语言下的集合强大，自然也比其它语言下的集合框架复杂。其中一个重要点就是它支持视图，集合对象可以创建视图对象，视图对象又可以继续创建更深层的视图对象，以此类推，可以创建出一颗以原始集合对象为根且深度无限的视图对象树，树上所有视图对象均共享原始集合的数据，对任何一个视图对象进行修改都能够影响到原始集合对象的数据。在其它语言的集合框架中，要么不支持视图，要么视图是只读的，要么仅在极个别场合支持视图，但在Java集合框架中，视图行为具备普适性。

请参考如下例子 

// 原始集合, { A: " Alpha" , B: " Beta" , C: " Gamma" , D: " Delta"  }
MANavigableMap<String, String> map = new MATreeMap<>();
map.put("A", "Alpha");
map.put("B", "Beta");
map.put("C", "Gamma");
map.put("D", "Delta");

// 第一级视图集合, { D: " Delta" , " C" : " Gamma" , B: " Beta" , A:" Alpha"  }
MANavigableMap<String, String> descendingMap = map.descendingMap();

// 第二级视图集合, { " C" : " Gamma" , B: " Beta"  }
MANavigableMap<String, String> subMap = descendingMap.subMap("C", true, "B", true);

// 第三级视图集合, [ " B" , " C"  ]
MANavigableSet<String> descendingKeySet = subMap.descendingKeySet();

// 第四级视图集合, [" B" ]
MANavigableSet<String> headSet = descendingKeySet.headSet("C", false);

// 第五级视图迭代器：可以迭代出元素”B”
MAIterator<String> descendingIterator = headSet.descendingIterator();

这段代码创建了6层对象的链式结构。第一层为原始集合对象，其余5层均为视图对象，视图对象并不持有真实数据，但可以共享原始集合的数据，这样，修改通过最深的一层视图对象descendingIterator也可以做出修改并影响到原始集合map得数据。如下 

descendingIterator.next(); //让迭代器定位到元素”B”
descendingIterator.remove(); //删除当前正在被迭代的元素”B”

此操作后，原始集合会受到影响，其数据会变化成{ A: “Alpha”, C: “Gamma”, D: “Delta” }。在此过程中，修改事件由最深一层视图对象descendingIterator率先触发，然后冒泡到上一层视图导致上一层视图触发一个新事件，以此类推，直到最顶层的原始集合本身触发事件通知为止，整个冒泡机制可以用下图描述。

通过集合视图链的视图冒泡机制，无论用户通过多么间接和隐晦的手段达到修改集合数据的目的，都能导致最顶层的原始集合本身触发修改事件。这就是ObjectModel4Java中的数据模型类敢于直接将集合属性暴露成公有属性的原因。

JPA部分

实际项目中，除了会使用纯内存结构的内存数据模型外，更有可能是JPA数据模型。JPA虽然善于在实体对象和数据库之间交互数据，但是忽视实体对象之间的关系的智能管理和双向关联一致性维护。所以从ObjectModel4Java扩展出了ObjectModel4JPA以支持智能JPA/Hibernate实体类，一方面负责为JPA实体类增加智能对象关系管理的能力，另外一方面扩展ORM框架本身使其更为强大。

ObjectModel4JPA

ObjectModel4JPA是ObjectModel4Java功能针对JPA实体对象的高级扩展，目的是为了兼容ORM领域特有的延迟属性。

ObjectModel4JPA和ObjectModel4Java的重要差异
  1. 由于受到JPA/Hibernate的限制，JPA字段无法声明成java.util.SortedSet，java.util.SortedMap，java.util.NavigableSet或java.util.NavigableMap类型，仅能声明成java.util.Set或java.util.Map。如果有这种需求，只能将字段声明成Set或Map类型并使用注解@org.babyfish.model.Navigable。
  2. 除非使用了Set类型的字段使用了注解"@org.babyfish.model.Navigble"，或者业务上有明确的需求，一般建议不要轻易地使用注解@org.babyfish.model.ComparatorRule为set或map类型的字段指定set元素或map值的的hashCode计算规则，因为默认的规则非常强大。默认规则如下
    1. 首先，ObjectModel4JPA中id(@javax.persistence.Id)属性不允许为基本类型，否则框架报错。
    2. 如果当前对象id为null，使用System.identityHashCode计算实体对象的hashCode，而equals则判断两个引用是否指向同一个实体对象。
    3. 如果当前对象id非null，使用id计算实体对象的hashCode，而 equals则判断两个实体对象的id是否相等。
    4. 如果实体对象的id被改变，由不稳定集合元素机制自动调整所有相关集合。请放心使用Hibernate的id生成策略和“hibernate.use_identifier_rollback”配置。
ObjectModelJPA实体对象之间的双向关联

JPA实体对象远远比简单的Java数据对象复杂，因为它支持延迟属性，如

  1. 针对一对多或多对多的延迟集合属性
  2. 针对多对一或一对一的延迟代理属性
  3. 针对非关系的属性的普通延迟属性，@Basic(fetch = FetchType.LAZY)

在JPA实体之间的双向关联中，必须一端属性为真实端，而另外一端为虚假端。

  • 例子一, "Employee.department"为真实端，"Department.employees"为虚假端。

    1. "Department"类的定义:

    private Set<Employee> employees;
}
    2. "Employee"类的定义: 
      @JPAModel
@Entity
@Table(name = "EMPLOYEE")
public class Employee {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "DEPARTMENT_ID") //真实端
    private Department department;
}

  • 例子二, "Department.employees"为真实端，"Employee.department"为虚假端。

    1. "Department"类的定义:

@Entity
@Table(name = "DEPARTMENT")

    @OneToMany(mappedBy = "department") 
    @JoinColumn(name = "DPERATMENT_ID") //真实端

在原始的Hibernate中，只有通过真实端修改对象关系才能引起数据库中关系的修改，如果真实端集合属性是延迟的，当它被修改时会立即加载数据，这对真实端的集合而言是必须的。当然，在延迟集合加载过程中，如果其内部session丢失，将会导致异常。

在原始的Hibernate中，对虚假端的修改时没有必要的，即使用户已经修改了虚假端的属性，也不能引起数据库中关系的修改。修改虚假端的延迟集合会导致因加载数据而带来的不必要的性能开销，如果在加载过程中发现其session丢失，还会导致不必要的异常。

ObjectModel4JPA具备和ObjectModel4Java完全相同的功能，为了保证双向关联的一致性，当一端被修改时，BabyFish会自动地修改另外一端, 这就意味中两端总是会被一起修改，数据库总会被更新。但是，这会导致不必要的查询浪费甚至是不必要异常吗?

为了解决这个问题，ObjectModel4JPA替换了Hibernate的延迟机制, 包括延迟集合，延迟代理和普通延迟属性。BabyFish提供了自己的Hibernate延迟集合框架，这是一个非常“狡猾”的集合框架，对于一个虚假端的未加载集合而言

  • 如果它被应用程序主动修改，那么它的行为和经典的"*.hbm.xml"文件中配置了lazy="extra"的集合属性极其类似, 除了会执行一条超小的SQL到数据库中去检查当前被操作元素的存在性外，并不会导致数据加载；但是本集合被修改的事件仍然会被“捏造”并触发，最终导致ObjectModel4会自动对另外的真实端属性进行修改，最终数据库关联修改会成功。同时它会记住修改历史以备和将来的延迟查询结果进行合并。此功能叫"延迟修改"(Lazy modification)
  • 如果它被ObjectModel自动修改, 这代表应用显方式修改的是另外一方的真实端，数据库关系必要会被修改。整个过程中，该集合并不会执行任何SQL，但本集合被修改的事件仍然会被“捏造”并触发, 同时它会记住修改历史以备和将来的延迟查询结果进行合并，此功能叫"虚假修改"(Dummy modification)

所以，对于使用了ObjectModel4JPA的JPA实体而言，使用者不用担心实体双向关联中哪一端是真实的哪一端是虚假的。无论修改哪一端，另外一端都会被自动“狡猾地”修改，最终提交事务时数据库中的关系种会被修改；同时也不用担心此举会引入性能问题。

立竿见影，分布式项目开发利器：QueryPath

QueryPath使JPA查询具备动态数据抓取能力和动态数据排序能力。它不但能避免让Hibernate生成性能低下的具有N+1查询问题的SQL；对于中大型的分布式应用而言，不仅如此，还能对让很多开发人员感到头疼的延迟加载失败异常的问题提供完美的解决方案。这也是本框架从Active Record和ADO.NET Entity Framework中借鉴并进一步发扬光大的最重要的功能。

QueryPath可以分为两个部分，FetchPath和SimpleOrderPath，我们先讨论FetchPath。

使用Hibernate开发实际项目，遇到的异常以org.hibernate.LazyInitializationException为最多，该异常表示因实体对象内部Hibernate Session的失效导致ORM实体对象上的延迟属性无法从数据库中加载真实数据。为了解决这个痛点，Spring支持了“Open Session In View”。

所谓Spring之“Open Session In View”，指对Web应用而言，在接收到HTTP请求时，服务端开启一个Hibernate Session并一直持续至HTTP响应完成，在这个过程中所有的Web组件，业务组件和数据访问组件均借助于该Hibernate Session运行。由长生命周期的Hibernate Session的存在，任何ORM实体对象上的延迟属性均可从数据库随意加载真实数据，而不会导致org.hibernate.LazyInitializationException异常。但这个做法有三个问题:

  • 该策略仅仅对传播机制为REQUIRED的内嵌事务友好。
  • 这会导致频繁地加载延迟属性，大量的N+1查询SQL会被生成，影响程序性能。
  • 最严重的问题，此策略仅仅能适用于单JVM实例的超简单应用，对于多JVM实例的分布式应用却无能为力。

假如现有一个分布式应用，业务层运行在应用服务器集群上，而Web层运行在Web服务器集群上，两个集群彼此独立，靠远程调用来彼此通信。在应用服务器集群中业务层的数据查询接口返回了一些带有未加载延迟属性的ORM实体数据，这些数据再远程地传递到Web服务器集群中，这个过程必须借助一些远程对象传递技术，如：RMI, SOAP或JSON等，无论采用何种技术作为远程对象传输协议，包含未加载延迟属性的ORM实体数据在被远程传递到目的地后，其内部持有的用于数据延迟加载的Hibernate Session均会丢失，最终Web集群拿到的ORM实体对象的未加载延迟属性不能正常地加载真实数据，只能抛出异常。

为了解决此问题，BabyFish框架支持了QueryPath。QueryPath是一种结构简单的、适合于远程传输的、可以由客户端动态指定的描述符。这些简单的描述符可以在发出数据查询的远程请求时作为查询参数的一部分从客户端远程传输至服务端，服务端最底层的数据访问组件在接受到这些动态描述符后，将它们解析成自己可以理解的延迟属性抓取计划并合并到最终生成的SQL语句中去，这样一口气查询出客户端所需要的一切数据对象图。最后这些数据对象图被远程返回给客户端，由于客户端所需要的所有数据均被预先查询，绝不存在自己需要但没有被加载的数据，所以客户端自然不需要加载延迟属性。现在以两幅图片来描述不使用QueryPath和使用QueryPath的区别。

QueryPath也支持对查询结果和集合关联属性进行动态排序，此即为SimpleOrderPath，示例代码如下 

// 调用查询方法，使用5条FetchPath和4条SimpleOrderPath
List<Department> departments = departmentRepository.getDepartmentsLikeName(
    "%s%",

    /*
     *这5条以begin开头以end结尾的QueryPath叫做FetchPath，同上例
     */
    Department__.begin().description().end(),
    Department__.begin().employees().image.end(),
    Department__.begin().employees().annualLeaves().end(),
    Department__.begin().employees().sickLeaves().end(),
    Department__.begin().company().end()

    /*
     * 这4条preOrderBy开头以asc或desc结尾的QueryPath叫做SimpleOrderPath
     */
    // 查询结果本身按部门的名称升序排序
    Department__.preOrderBy().name().asc(),

    // 对于每个部门，其集合属性”employees”是排序set, 按照员工的名称升序排序
    Department__.preOrderBy().employees().name().asc(),

    // 对于部门的每个员工，其集合属性”annualLeaves”是排序set, 按照年假的开始时间降序排序
    Department__.preOrderBy().employees().annualLeaves().startTime().desc(),

    // 对于部门的每个员工，其集合属性”sickLeaves”是排序set, 按照病假的开始时间降序排序
    Department__.preOrderBy().employees().sickLeaves().startTime().desc()
);

此例子在调用数据查询方法时一共使用了9条QuryPath，其中有4条SimpleOrderPath，第一条SimpleOrderPath对查询结果本身进行排序，这很好理解；但是后面的3条SimpleOrderPath均是对集合属性的排序，而在此例所用的JPA实体类中，集合属性均为set，这是怎么回事呢？原因很简单，在ObjectModel(无论是ObjectModel4Java还是ObjectModel4JPA)中，除非set或map字段被注解@Navigale修饰，或其接口类型是java.util.NavigableSet或java.util.NavigableMap，默认的set和map都能记忆数据的添加顺序，这点非常类似于java.util.LinkedHashSet或java.util.LinkedHashMap。顺序难以预料的普通set和map绝对不会被ObjectModel所使用。

强类型QueryPath固然好，但在实际项目中，未必在所有情况下都比较容易指定强类型的QueryPath。比如，在Web页面上一个表格支持用户通过点击表头上的列标题来指定按照某一列升序或降序排序，这时，需要排序的属性名和排序方式都是Web层的JavaScript脚本以字符串请求参数的方式发送到服务端的，这时可以使用弱类型QueryPath，如下面这段Web服务端代码所示（其中request为javax.servlet.HttpServletRequest类型）。 

// 取出Ajax请求中Javascript传递的用来排序的属性的名称。
String propertyName = request.getParameter("propertyName");

// 取出Ajax请求中Javascript传递的排序的方式，必须为”asc”或”desc”。
String orderMode = request.getParameter("orderMode");

// 拼接出弱类型的QueryPath字符串。
String queryPathString = "pre order by this." + propertyName + ' ' + orderMode;

// 将弱类型的QueryPath字符串编译成强类型的QueryPath对象。
Department__[] queryPaths = Department__.compile(queryPathString);

针对Oracle，克服Hibernate分页查询缺陷

本节即将论述，Hibernate的分页查询存在着难以解决的缺陷。但可喜的是，针对数据库是Oracle的情况，BabyFish框架能解决此缺陷，对用户唯一的要求是不能使用Hibernate提供的Oracle方言org.hibernate.dialect.Oracle10gDialect；而要使用经过BabyFish扩展的Oracle方言org.babyfish.hibernate.dialect.Oracle10gDialect。

先来看一个简单的JPA查询的例子（假设this.em是javax.persistence.EntityManager类型） 

return this
        .em
        .createQuery("select d from Department d order by d.name asc")
        .setFirstResult(100) // 分页查询：跳过前面100条记录
        .setMaxResults(10) // 分页查询：最多查询10条记录
        .getResultList();

以Oracle为例，Hibernate生成的SQL如下

select 
    * 
from ( 
    select 
        row_.*, 
        rownum rownum_ 
    from ( 
        select
            d.DEPARTMENT_ID, d.NAME, d.COMPANY_ID 
        from DEPARTMENT d 
        order by d.NAME asc 
    ) row_
    where rownum <= ?
) 
where rownum_ > ?

这是一句经典的基于RONUM伪列的Oracle分页查询，到目前为止，似乎一切良好。但是，如果我们改一下Java代码，如下

return this
        .em
        .createQuery(
                "select d from Department d" +
                "left join fetch d.employees " + //对集合属性employees进行fetch
                "order by d.name asc"
        )
        .setFirstResult(100) // 分页查询：跳过前面100条记录
        .setMaxResults(10) // 分页查询：最多查询10条记录
        .getResultList();
再试一次，会发现生成的SQL如下 
Select
    d.DEPARTMENT_ID, d.NAME, d.COMPANY_ID,
     e.EMPLOYEE_ID, e.NAME, e.DEPARTMENT_ID
from DEPARTMENT d 
left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
order by d.NAME asc
我们发现，这次生成的SQL中不带任何分页查询，同时还能发现，Hibernate的日志打印了一句警告。
firstResult/maxResults specified with collection fetch; applying in memory!

当查询包含对集合属性的fetch行为时，Hibernate不得不放弃在SQL层面分页，将所有数据全部查出并映射成内存对象，最后在内存中完成分页。这是一个巨大的性能隐患，而且这样的警告日志很难引起开发人员的重视。

为什么Hibernate会这样做呢？原因是集合属性的join抓取会导致父表数据在查询结果中的重复，请看如下查询结果：

父表"DEPARTMENT"的数据 子表"EMPLOYEE"的数据
DEPARTMENT_ID NAME EMPLOYEE_ID NAME DEPARTMENT_ID
1 Market 1 Jim 1
1 Market 2 Kate 1
2 Sales 3 Lucy 2
2 Sales 4 Tom 2
如表格所示，在连接查询结果中，左边的父表数据出现了重复，用户关注的查询对象就是左边的Department，而不是整个连接结果，join连接仅仅是为了抓取Department.employees属性而已。显然，这时对查询结果进行SQL分页操作其实并不是对用户关心的左侧DEPARTMENT数据的分页行为。

幸运的是，对于Oracle数据库而言，如果开发人员使用BabyFish框架提供的方言org.babyfish.hibernate.dialect.Oracle10gDialect，再试一次，会发现生成的SQL如下。 

select 
    *
from (
    select
        d.DEPARTMENT_ID, 
        d.NAME, 
        d.COMPANY_ID,
        e.EMPLOYEE_ID,
        e.NAME,
        e.DEPARTMENT_ID,
        dense_rank() over(/* DENSE_RANK is built-in analysis function of Oracle.*/
            order by 
                d.NAME asc, /* 原始JPQL中的排序字段，未必唯一 */
                d.ROWID asc /* BabyFish追加的排序字段，必然唯一 */
        ) dense_rank____
    from DEPARTMENT d 
    left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
)
where dense_rank____ > ? and dense_rank____ <= ?

这里，BabyFish使用Oracle内置的分析函数DENSE_RANK先对连接结果中左边父表数据进行去重排名，然后再根据去重后的名次进行分页。幸运的是，由于原始的order by语句仅仅对DEPARTMENT表本身的字段进行排序，所以DENSE_RNAK分析函数能够胜任。

接下来，看看最不幸的场景，再改Java代码如下

return this
        .em
        .createQuery(
                "select d from Department" +
                "left join fetch d.employees " + /* 对集合属性进行fetch */
                "order by " +
                "  d.name asc, " + /* 不但要对DEPARTMENT表的字段排序 */.
                "  e.name asc" /*, 还要对EMPLOYEE表的字段排序*/
        )
        .setFirstResult(100) // 分页查询：跳过前面100条记录
        .setMaxResults(10) // 分页查询：最多查询10条记录
        .getResultList();

select 
    *
from (
    select
        d.DEPARTMENT_ID,

                d.NAME asc, 
                e.NAME asc

    from DEPARTMENT d 
    left outer join EMPLOYEE e on d.DEPARTMENT_ID = e.DEPARTMENT_ID
)
