org.babyfishdemo.foundation.typedi18n In classic solution, you can do the I18N like this:
  • Create some resource files: such as resourceName.properties, resourceName_en.properties, resourceName_zh_CN.properties, resourceName_fr_FR.properties.
  • Each resource file has some key-value pairs, the keys of all the resource files must be same, the values should be wroten by different natural languages. Optionally, value can contain some parameters such as {0}, {1}, {2} ...
  • Use "java.util.ResourceBundle.getBundle" to create bundle instance to load one of those resource files by auto/mannual locale choosing, then invoke the "getString" by using the key declared in the resource file to be the parameter to get the I18N string. Optionally, the parameters of the I18N must be replaced.
Like script languages, the solution is a weak-type solution, it's not easy to avoid the spelling mistakes, for examples
  • The illegal resource name is specified when the "ResourceBundle.getBundle" is invoked
  • The illegal key is specified when the "ResourceBundle.getString" is invoked
  • Optionaly, when the parameters of I18N string is replaced by java code, it's not easy to guarantee the text replacement count is same with the actual parameter count of the value of resource.
Worse still, there are so many weak-type dependencies so that the progam is very hard to be refactored.
The demo under this package will demonstrate how to resolves these problems.
AnnualLeaveRequestTest.java Under the source folder src/main/ava, the class org.babyfishdemo.foundation.typedi18n.AnnualLeaveRequest supports many static native method that are marked by the annotation @org.babyfish.lang.I18N.
  • The names of these method must be same with the keys of the resource file org/babyfishdemo/foundation/typedi18n/AnnualLeaveRequest.properties.
  • The parameter count of any one of these methods must be same with the parameter count of the corresponding value of the resource file org/babyfishdemo/foundation/typedi18n/AnnualLeaveRequest.properties.
  • If a value of resource file has parameters, "{0}" must exists
  • If a value of resource file has parameters, the parameter indexes must be sequential(For example: {0}{1}{2}{1}{0} is allowed but {0}{2} is illegal)

All the restrictions will be validated at compilation-time, if there are some errors, compilation errors will be reported; otherwise, the bytecode or the class org.babyfishdemo.foundation.typedi18n.AnnualLeaveRequest will be changed, the native flag of these methods will be cleared and their bytecode will be added to invoke the I18N api of java.util.ResourceBundle.

This demo only has one resource file org/babyfishdemo/foundation/typedi18n/AnnualLeaveRequest.properties, actually, you can create other resource files such as "AnnualLeaveRequest_zh_CN.properties".

org.babyfishdemo.xcollection.basic
EqualityComparatorTest.java

As we know, "java.util.TreeSet<E>" and "java.util.TreeMap<K, V>" support two ways to specify the element comparison rule

  • Let the element/key class itself implement the interface "java.lang.Comparable"
  • Use another class who implements the interface "java.util.Comparator" to specify the 3rd-party comparison rule

But, the other sets or maps only can let the element/key class itself override the method "int hashCode()" and "boolean equals(Object)", no 3rd party comparison rule is allowed to specify how to calculate the hashCode of element/key and how to check whether two elements/keys are equal.

In order to resolve this problem, X Collection supports org.babyfish.collection.EqualityComparator<T> to specify the 3rd party comparison rule about hashCode calculation and equality checking.

ReplacementRuleTest.java

In Java collection Framework, set does not allow duplicated elements, map does not allow duplicated keys.

If there is an set, it contains an element, let's call it "a", now a new element "b" which is equal to "a" but isn't same with "a" is added to the set, the size of that set won't be changed and the method "add" will return false absolutely. but there is another question, should the element "a" of that set be replaced to "b"?

In order to resolve this problem, the set/map of X Collection Framework support an initial argument whose type is org.babyfish.collection.ReplacementRule. It can specify whether the existing element/key of the set/map should be replaced by the conflict new element/key or not.

org.babyfishdemo.xcollection.unstable

The package demonstrates "Unstable Collection Elements" which is the most important functionality of X Collection Framework.

As we know, Java Collection Framework supports hash structure(HashSet, HashMap, LinkedHashSet, LinkedHashMap) and red black tree structure(TreeSet, TreeMap) that provide high searching preformance. The element/key cannot be changed after it has been added into hash structure or red black tree structure; otherwise, the hash structure and red black tree struture will be broken so that their behavior will become unpredictable.

In order to resolve this problem, X Collection Framework supports org.babyfish.collection.FrozenEqualityComparator<T> for hash structure, and supports org.babyfish.collection.FrozenComparator<T> for red-black tree structure. These comparators can let the sets and maps support "Unstable Collection Elements". An object can still be modified after it has already been added into sets as element or added into maps as key, the internal structure of corresponding sets and maps will be adjusted automatically.

org.babyfishdemo.macollection.batch

Java Collection Framework supports some batch modification methods, such as

  • java.util.Collection<E>.addAll(java.util.Collection<? extends E> c);
  • java.util.Collection<E>.clear();
  • java.util.Collection<E>.removeAll(java.util.Collection<?> c);
  • java.util.Collection<E>.retainAll(java.util.Collection<?> c);
  • java.util.List<E>.addAll(int index, java.util.Collection<? extends E> c);
  • java.util.Map<K, V>.putAll(Map<? extends K, ? extends V> m);
  • java.util.Map<K, V>.clear();

When the developer invokes any one of these methods, several collection elements will be changed. Each changed element can trigger a couple of events, each couple is combined with a pre event which is raised before the modification and a post event which is raised after the modification.

Developers use their event listener implementation to determine how to handle the events, of course, exception is allowed to be thrown in their event listener implementation. In order to let the behavior of Java Collection Framework batch modification is reasonable, MA Collection Framework guarantees

  • No elements will be changed if any one pre event listener implementation throws exception
  • If no pre event listener implementation throws exception, all the elements that should be changed even if some post event listener implemenations throw exception

That means batch modification for collection will be fully successful or fully failed, it's absolutely impossible to cause certain elements are modified while other elements have not been modified.

org.babyfishdemo.macollection.bubble

Many interfaces of Java Collection Framework supports view creating methods, such as "java.util.List.subList(int, int)", "java.util.Map.keySet()", "java.util.NavigableMap.subMap(K, boolean, K, boolean)", "java.util.Collection.iterator()", etc. Of course, it's possible to create deeper view on view too.

Finally, the event will be bubbled to the root collection and a new event will be triggered on the root collection.

ReadWriteLockSupportingTest.java

Like "java.util.Collections.synchronized...()", "MACollections" supports some static methods to create thread-safe wrapper proxy for non-thread-safe collection.

"MACollections.locked(...)" dose not use the java "synchronized", it returns the thread-safe proxy base on read-write lock "java.util.concurrent.locks.ReadWriteLock", so that the readonly methods of the returned proxy only acquires the read lock.

In order to support read-write lock, the original collection must implement "org.babyfish.lang.LockDescriptor" and let its method "isReadWriteLockSupported()" return true if it's possible. Otherwise, that means the original collection does not support read-write lock and "MACollections.locked(...)" returns an proxy with methods that only acquires write lock.

LockPropagationTest.java

Sometimes, the lock scope of thread-safe collection is too small, For example, "java.util.Collections.synchronized...(...)" can return thread-safe wrapper but the iterator of the wrapper is not thread-safe, it requires the programmer to mannually acquire the lock to protect the loop of iterator, but this work is easily to be forgot.

In order to resolve this problem, BabyFish supports "Lock-Propagation". For thread-safe proxy, it allows the programmer to enlarge the lock scope.

Be different with "java.util.Collections.sysnchronized...(...)" that returns a thread-safe proxy whose iterator must be protected by the programmer manually. "org.babyfish.collection.MACollections.locked(...)" returns a thread-safe proxy whose "iterator()" throws an exception to tell the user the iterator of locked proxy can only be used in the "MACollections.locking(...)".
PrimitiveListTest.java Demonstrates how to wrap the primitive array to list. Its functionality is same with "java.util.Arrays.asLsit(T...)", but it accept the array whose component type is primitive type.

In java language, the generic type parameter can not be primitive, so that "java.util.List<Ineger>" is valid but "java.util.List<int>" is invalid. In classic solutions, we have to translate "int[]" to "Integer[]" and then use "java.util.Arrays.asList". All the "int" is translated to "Integer" objects, If some of those objects are moved to old generation(or region) of GC heap, that is not friendly for GC.

Babyfish supports "MACollections.wrapXXX(...)" that can wrap the primitive array to collection directly, the collectionn does not translate "int" to "Integer" and it retains the primitive array. Finally, those "int" elements will be translated to "Integer" objects when programmer use iterator to visit them. It is not bad becuse those objects are created in the youngest generation(or region) of GC heap and are collected very fast if they are lost soon.

EntrySetProtectionTest.java Demonstrate the method "setValue" of java.util.Map.Entry<K, V> can be forbidden if the entry set is created by "MACollections.undifiable(map.entrySet());" (Classic solution can only do this by "Collections.unmodifiableMap(map).entrySet()")

Embeddable model class cannot be used by other association property, it can only be used by scalar property(or be the key type of Map association). For a scalar property, its type can be an embeddable model class, of course, the embeddable model class can have its scalar property whose type is another nested embeddable model class, That means ObjectModel allows the developer to declare embeddable property chain.

When simple type scalar property of the deepest embeddable model object is changed by the programmer, the scalar changing event will be raised to let the deepest embeddable model object know it's changed, then the event will be bubbled to let the parent embeddable model object know it's changed too. Finally, the root model object whose type is "@org.babyfish.model.ModelType.REFERENCE" can feel it is changed. This is the functionality "Bubbled Scalar Event", it looks like the functionality "Bubbled Event" of MA Collection Framework very much.

babyfishdemo-jpa-criteria

This project demonstrates the brand new JPA Criteria implementation of BabyFish. which is used to replace the implementation of Hibernate.

  • In order to add some new functionalities to JPA Criteria, Babyfish supports a new interface org.babyfish.persistence.criteria.XCriteriaBuilder that extends javax.persistence.criteria.CriteriaBuilder.
  • Most demos about JPA functionalities choose to query data by Criteria API, Not by JPQL(or NamedQuery, Spring-Data-JPA), becuase
    • Be different with the JPQL, The JPA Criteria implemenation of BabyFish has some performance optimization functionalities.
    • For java language, JPQL is string literal, all errors of JPQL can only be reported at runtime. It's not easy to guarantee the JPQL is correct before running or testing the application so that it's need more time to debug the application, though JPQL needs less characters for code.
    • In practical projects, when entity class is refactored, Criteria API can report compilation errors immediately until the refactor is finished, but JPQL can only report the errors at runtime by unit test(or integration test) so that it's dangerous for entity class refactoring, because the test coverage rate might not be 100% and fixing runtime errors is harder than fixing compilation error.
      Refactor efficiency is more important than the writing efficiency. especially for big projects.

testLiteral

Hibernate translates numbric literal to SQL literal because Hibernate only thinks numberic literal can not cause "SQL Injection Attacking" problem, but it will reduce the hit rate of Hibernate's HQL QueryPlan cache and database's SQL QueryPlan cache.

In babyfish, "javax.persistence.criteria.CriteriaBuilder.literal()" is implemented by Open-JPA style: Literal Expression will always be transalated to SQL parameters. It's very helpful for both HSQL query plan cache of Hibernate and SQL query plan cache of Database.

In practical projects, the user only need to use "javax.persistence.criteria.CriteriaBuilder.literal()", Please say "Good Bye" to the method "javax.persistence.criteria.CriteriaBuilder.parameter()"
testConstant Some databases, such as Oracle, support "function index", BabyFish supports "org.babyfish.persistence.criteria.XCriteriaBuilder.constant()" to create constant expressions, be different with literal expressions, constant expressions will always be translated to SQL literals. Crating function index for any complex expressions that contain SQL those constant literal expressions can otpimize the performance.

babyfishdemo-paging-query This project demonstrates the paging query of BabyFish, the most important two functionalities are
  • Unlimited count:

    Paging Query is popular in practical projects, it requires two SQL statements

    • SQL to query the total row count.
    • SQL to query the rows in this page.

    These two SQL statements are different, but there are some similarities between them. When one is changed, the other one must be changed too, this work is boring and error-prone.

    BabyFish supports a functionality called "UnlimitedCount", the programmer only need to create the query object by the second SQL statement, then invoke the method "org.babyfish.persistence.XQuery.getUnlimitCount()" of that query object, this method call internally generate the first SQL statement and return the total row count.

  • DistinctLimitQuery: This functionality is Oracle only.

    Hibenrate has a big performance issue. When a paging query contains collection fetches, hibernate will use memory-level paging query: query all the rows from database and do the paging filter in memory.

    BabyFish supports a functionality called "DistinctLimitQuery", When database is Oracle, the paging query will always be applied on database level even if the query contains collection joins.

HsqldbMemoryLevelLimitQueryForDistinctTest.java

When query type is "org.babyfish.persistence.QueryType.DISTINCT", and the JPQL contains the collection fetches, what will happend?

Hibernate has an issue, for the query with simple JPQL, it applies the paging query at the SQL level, but, if the JPQL contains collection fetches, it will query all the matched rows form database and then applies the paging filter in the memory, this is big performance problem.

  • When database base is Oracle, this performance issue can be resolved by BabyFish, this functionality will be demonstrated by next demo class.
  • For other databases, BabyFish have no idea too, it has to keep the default behavior of Hibernate, applies the pagging query in memory. Hibernate also thinks it's dangerous behavior so that it will generate a warning message in the log file about the paging query has been applied in memory, but babyfish thinks that's not enough to draw the developer's attention, it's so easily to be ignored.
A little difference with Hibernate.

  • As default behavior, the memory level paging query is disabled, an exception will be raised when the JPQL with both paging restriction and collection fetches is executed.

  • The programer must set the jpa/hibernate property "babyfish.hibernate.enable_limit_in_memory" to be true to enable the memory level paging paging if the he/she is willing to accept this performance issue and still want to use the memory level paing query.

    Please see
    <property name="babyfish.hibernate.enable_limit_in_memory" value="true"/>
    of src/test/resources/persistence.xml
